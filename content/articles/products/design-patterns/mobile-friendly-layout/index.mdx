モバイル対応をする基準と、その対応方法を定義します。

SmartHR において、従業員アカウントのモバイルからの利用は全体の7割以上を占めます。業種や業態に依っても異なりますが、業務中にデスクトップを使わない利用者も多いためモバイルの考慮は必須です。


## モバイルレイアウトの定義

「モバイルレイアウト」とは、スマートフォンなどのモバイルでの利用を想定してUIを設計・開発することを指します。

ユーザーエージェントではモバイルかどうかを判別しきれないため、**画面幅が751px以下の場合をモバイルとし、モバイルレイアウトの適用対象とします。**

### レスポンシブとアダプティブの使い分け

本ドキュメントでは以下のように使い分けます。

- レスポンシブ： 同じ構造・データ、見た目を保ちつつ、画面幅に応じてレイアウトを変えること
  - 例： 画面幅に依って文章やボタンが折り返される
- アダプティブ： 画面幅や端末、利用環境に応じて、構造やデータ、見た目を変えること
  - 例； モバイルでは情報量を減らし、操作を簡略化する
- レスポンシブ/アダプティブデザイン： レスポンシブ/アダプティブに設計すること
- レスポンシブ/アダプティブレイアウト： レスポンシブ/アダプティブに設計されたUIのこと


## 基本的な考え方

### 複雑な操作を避けよう

モバイルでの利用は、移動中や空き時間などデスクトップに比べて短い時間での利用が多いです。デスクトップ向けに設計されたUIをレスポンシブに再レイアウトするだけでなく、情報の取捨選択や簡潔な表現の徹底が求められます。

モバイルにおける従業員向け機能では、基本的に1画面で複数の操作をさせるべきではありません。デスクトップ向けに最適化されたUIをレスポンシブデザインで対応するには限界があるため、アダプティブに設計する必要があります。


## 設計基準

### モバイルでは複数カラムにしない

複数カラムレイアウトを避けましょう。画面の幅によってコンテンツの可読性を保てなくなります。代わりに縦に積み上げるか、複数の画面に分けるなど、1度に表示する情報量を減らしましょう。

縦に積み上げて解決ができない場合は、アダプティブなレイアウトを検討してください。

### 読み方向のスクロールや2次元スクロールを避ける

読み方向のスクロールや2次元スクロールは、コンテンツを読み理解するために必要な労力を著しく増やすので、避けましょう。画面幅からコンテンツがあふれる場合は、折り返しましょう。

ただし、画像や動画・地図など2次元であることが情報を伝える上で重要な場合は、その限りではありません。

### 余白の取り方

デスクトップでは[代表的なページ内レイアウトのパディング]()に則り32pxだが、モバイルでは表示コンテンツ量を増やしたいため、水平方向のパディングを16pxとする。

### Buttonの大きさとレイアウト

モバイルでは基本的に通常サイズの[Button]()を使ってください。操作ミスが増えるため小サイズを使わないでください。

画面内で主要な操作を置く場合は、コンテンツ幅いっぱいのボタンを使ってください。

プライマリーボタンとセカンダリーボタンなど複数の操作を水平方向に並べる場合は、識字方向の後方にプライマリーボタンが来るようにレイアウトしてください。LTR（Left To Right）では、右側にプライマリーボタンを置きます。

### コンポーネントごとの使い方

SmartHR UI は基本的にレスポンシブに実装されているため、特出して触れる必要のあるコンポーネントについて記述します。

#### useDevice（検討中）

`useDevice` フックを使って `isNarrowScreen` を取得し、必要に応じてコンポーネントに割り当ててください。SmartHRにおける頻出パターンはSmartHR UIでアダプティブなレイアウトを実装予定です。各プロダクト側でユーザーエージェントの判定を実装する必要はありません。

#### Dialog

SmartHR UI 側で実装するため `isNarrowScreen` を渡してください。基本的にプロダクト側で考慮することはありません（実装まだ）。

#### BottomFixedAreaとFloatArea

そのままレスポンシブに使うと高さを取りすぎてしまうため、モバイル向けに再設計が必要です。[1画面で複雑な操作をさせない]()ように作ってください。

SmartHR UI でも最低限のレイアウトを実装するため `isNarrowScreen` を渡してください。

#### Pagination

モバイルでは基本的に使わないでください、より最適な方法があります。「表示を増やす」ボタンを領域の下部に置くなど、無限に読み込む機構を検討してください。

#### SideNav

複数カラムを前提としてレイアウトになってしまうため、モバイルでは使わないでください。Selectを使うなど、別の切り替え機構を検討してください。

#### Table

モバイルでは、画面幅を越えたテーブルは[2次元スクロール]()を招くため、縦積みのリスト形式を推奨します。

SmartHR UI 側で `isNarrowScreen` を用いたレイアウトの切り替えを検討中です。

#### Tooltip

画面領域が限定される中でTooltipを使って情報を押し込むのは現実的ではないでしょう。モバイルでは必要な情報を整理し、Tooltipは避けましょう。
